# RAG 调试台演示脚本

适用页面：

```text
http://127.0.0.1:8504
```

启动命令：

```powershell
cd "D:\new things\项目1\day1"
.\.venv\Scripts\streamlit.exe run rag_debug_app.py --server.port 8504 --server.fileWatcherType none
```

## 总体讲法

这个调试台不是给真实客户看的，而是给开发、面试和排查用的。客户侧页面只展示自然客服回复；内部调试台会把 RAG 链路拆开：先做问题分类，再做 Top-K 检索，再生成追问和工单草稿，最后才进入大模型回答。

建议面试时先不勾选 DeepSeek，先讲清楚分类、检索、metadata 和工单 JSON。讲完链路后，如果时间允许，再勾选生成最终回答。

## 演示一：售后边界与风险控制

问题：

```text
电器件通电了还能换货吗？
```

讲法：

这条用来展示高风险售后问题。系统不是直接承诺能换，而是先识别为售后质保，再从售后政策文档里找到电器件通电后的处理边界，最后生成给售后专员的工单草稿。

观察点：

- 问题类型应为售后质保。
- `needs_follow_up` 应为 `true`。
- Top-K 应命中《售后质保与退换货政策》。
- metadata 里 `category` 应为售后政策，`risk_level` 应为高。
- 工单草稿应要求补充订单号、购买时间、问题照片/视频、是否安装使用。

面试总结句：

这类问题最怕 AI 直接承诺退换货，所以我把售后规则作为高风险知识，并让系统先追问和转人工确认。

## 演示二：故障诊断与维修协同

问题：

```text
液压泵异响怎么排查？
```

讲法：

这条用来展示故障诊断场景。系统会把问题识别为故障类咨询，从故障诊断 FAQ 中检索液压泵异响的排查步骤，并把需要补充的油液、滤芯、视频和维修师傅判断整理进工单。

观察点：

- 问题类型应为故障诊断。
- Top-K 应命中《故障诊断FAQ》或液压件售后边界。
- 回答不应直接判断配件质量问题。
- 工单应收集故障视频、液压油状态、滤芯更换记录、维修师傅判断。

面试总结句：

我没有让客服 Agent 直接当维修师傅，而是让它做第一轮信息收集和风险分流，把复杂判断交给技术人员。

## 演示三：型号适配与缺信息追问

问题：

```text
没有零件号能不能配液压泵？
```

讲法：

这条用来展示适配规则。液压泵不能只按机型粗略匹配，系统会命中型号适配规则，提醒需要旧件铭牌、整机铭牌、总成照片或零件号，再进入人工确认。

观察点：

- 问题类型应为型号适配。
- Top-K 应命中《型号适配与件号确认规则》。
- 追问应包含机型、零件号、铭牌照片或旧件照片。
- 工单应把缺失信息列清楚，避免销售直接报价导致错配。

面试总结句：

这个例子能说明我把 RAG 用在业务风控上，而不是只做知识问答；系统会主动防止错配。

## 演示四：报价档位与销售策略

问题：

```text
你们能不能给三档方案？
```

讲法：

这条用来展示销售咨询。系统会检索报价档位说明或客户 FAQ，把方案拆成经济型、稳定型、原厂型，并提醒客服结合工况、预算和质保要求确认。

观察点：

- 问题类型应偏向报价咨询或配件报价。
- Top-K 应命中《报价与品质档位说明》或《客户常见问题FAQ》。
- 回答应强调三档方案的使用场景，而不是只给一个价格。
- 如果缺少具体配件型号，系统应继续追问机型、配件名和预算。

面试总结句：

我把销售话术结构化成可检索知识，Agent 可以按客户预算和使用场景输出更稳定的报价沟通路径。

## 演示五：库存咨询与采购跟进

问题：

```text
PC200液压泵有没有现货？
```

讲法：

这条用来展示库存类咨询。系统会识别客户在问现货，但不会编造库存数量，而是提示库存变化快，需要确认具体件号、品质档位和收货地，再由销售或采购跟进。

观察点：

- 问题类型应偏向库存或采购咨询。
- Top-K 可命中库存、报价、型号适配相关片段。
- 回答不能编造现货数量或发货承诺。
- 工单草稿应把具体机型、配件和收货地列为待补充信息。

面试总结句：

库存是强时效信息，不能靠 RAG 乱答。我会让 Agent 收集关键字段，并把最终确认交给销售或采购。

## 推荐演示顺序

1. 先演示“电器件通电了还能换货吗？”：最容易讲清楚风险边界。
2. 再演示“没有零件号能不能配液压泵？”：说明主动追问和防错配。
3. 再演示“液压泵异响怎么排查？”：说明故障诊断和转人工。
4. 时间够的话补“三档方案”和“现货咨询”：说明销售和库存场景。

## 一句话收尾

这个项目的重点不是做了一个聊天框，而是把客服问题拆成“分类、检索、追问、工单、回答”的流程，让 RAG Agent 能被观察、能被评测，也能被业务人员接手处理。
